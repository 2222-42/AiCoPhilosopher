# AI Co-Philosopher Constitution

## Core Principles

### I. Core Independence & Local-First Privacy

The system MUST be fully self-contained and operable without any external agent orchestration layers (e.g., Hermes Agent, OpenCode Go). External layers SHALL be treated strictly as optional enhancers, never as hard dependencies. The system MUST gracefully degrade and remain fully functional when external layers are unavailable or disabled.

User data, research projects, and all generated content MUST remain private and stored locally by default. The system MUST NOT transmit project content, living-document text, notes, hypotheses, or uploaded files/PDF contents to any external service unless the user has explicitly approved sending that specific content to a named external service for a specific request. When using external literature-search services, the system MUST transmit only the minimum necessary user-approved search data (e.g., query terms, author names, date ranges).

Rationale: Philosophical research often involves sensitive or unpublished ideas. Local-first operation guarantees intellectual privacy and ensures the system is always available regardless of external connectivity or third-party service status.

### II. Philosophical Accuracy & Intellectual Honesty

The system MUST prioritize philosophical accuracy, nuance, and intellectual honesty over speed or convenience. Every major claim or argument generated by any agent MUST include a confidence score (0.0–1.0) and a brief justification. The user SHALL always have final approval on any content that is permanently added to the living document.

The system MUST preserve a complete history of hypotheses, including failed paths and abandoned lines of thought. It MUST emulate the workflow of a human philosophical collaborator: ideation → literature review → concept analysis → argumentation → critique → synthesis → writing.

Rationale: Philosophy demands rigor. An AI collaborator that hides uncertainty or erases failed reasoning is intellectually harmful. Transparency builds trust and supports genuine research progress.

### III. Code Quality & Maintainability

All code MUST be clear, readable, and maintainable, following PEP8 (for Python) or the idiomatic style of the chosen language. Use descriptive names for variables, functions, classes, and modules. Avoid code duplication; prefer modular, reusable components. All public functions and classes MUST have docstrings or documentation comments. Use type annotations wherever possible. Handle errors gracefully and log failures for debugging and auditability.

Rationale: This is a long-running research tool. Poor code quality accumulates technical debt that directly impedes philosophical work. Maintainability ensures the system can evolve alongside the research it supports.

### IV. Testing Standards & Determinism

All core logic MUST be covered by automated tests (unit and integration). Tests MUST be deterministic and reproducible. Use mocks/stubs for external services in tests; the core test suite MUST NOT require network access. All bug fixes MUST include a regression test. Test coverage MUST be measured and reported for every release.

Rationale: Deterministic tests guarantee that philosophical reasoning components behave predictably. Network-independent tests ensure the local-first promise is verifiable in any environment.

### V. MVP-First Delivery & Continuous Improvement

Prioritize delivery of a working MVP that meets the minimum requirements: Project Coordinator, Literature Search Agent, Synthesis Agent, persistent workspace, and terminal interface. Defer non-essential features and optimizations until after the MVP is stable. The MVP MUST be fully functional without any external orchestration layers. All MVP features MUST be documented with clear usage instructions.

Encourage regular code reviews and refactoring. Solicit user feedback early and often to guide improvements. Maintain a changelog and versioning for all releases.

Rationale: Philosophical research tools are only useful if they exist and run. An imperfect but working tool delivers value immediately; a perfect but unfinished tool delivers none.

## Architecture Constraints

- The core system SHALL be implemented in Python using LangGraph (or an equivalent stateful multi-agent framework) as the default orchestration layer.
- The system MUST support an Adapter Pattern so that external orchestration layers can be plugged in without modifying core logic.
- The system MUST use open-source components and be fully self-hostable.
- All LLM outputs MUST be validated against Pydantic schemas before being accepted.
- Failed tool calls SHALL be retried up to 3 times with exponential backoff, then escalated to the Project Coordinator.

## Development Workflow & Quality Gates

- All PRs and reviews MUST verify compliance with this constitution.
- Complexity must be justified; simpler alternatives MUST be considered and documented.
- Progressive disclosure is mandatory: every coordinator response MUST start with a concise summary, followed by optional detailed sections.
- All agent decisions and tool uses SHOULD be logged for auditability and debugging.
- The External Agent Bridge MUST implement automatic fallback to internal LangGraph execution if the external layer is unavailable or returns an error.

## Governance

This constitution supersedes all other practices. Amendments require:
1. Documentation of change rationale.
2. Explicit approval.
3. A migration plan if the change affects existing workstreams or project state.

Versioning policy (semantic versioning):
- MAJOR: Backward-incompatible governance changes, principle removals, or redefinitions.
- MINOR: New principle/section added or materially expanded guidance.
- PATCH: Clarifications, wording fixes, typo fixes, non-semantic refinements.

Compliance review is expected at each release. The core system MUST never depend on external layers for basic operation.
